Tomcat靠堆砌操作系统线程来顶并发

Tomcat 的潜意识里有几个假设：
- 资源是无限且独占的： 物理机常年开着，我（Tomcat）启动慢一点（花 5 秒、10 秒）没关系，因为我一年到头也重启不了几次。
- 内存可以尽情挥霍： 物理机有的是内存，所以我启动时疯狂扫描所有的 Jar 包、预先创建几百个线程（每个线程吃 1MB 内存）常驻在内存里，我觉得理所当然。

**Tomcat 线程模型的局限（重）：** Tomcat 是 **Thread-per-request（一个请求一个操作系统内核线程）** 模型。一个物理线程初始就要 1MB 内存。在高并发或者 AI Agent 这种超长耗时的阻塞请求下，线程池会迅速耗尽，内存飙升。

_演进：_ 现代架构改用 **Netty（异步非阻塞）**、**Go 语言协程（Goroutine）**，利用单线程事件循环（Epoll 机制）或极轻量的协程（只占几 KB），用极低的资源消耗顶住十几万连接。

**冷启动阻碍了 Serverless（慢）：** 传统 Tomcat 启动时要高频扫描类路径、通过反射加载 Bean，耗时数秒。而 Serverless 追求极致的按量付费、瞬间拉起、用完即毁。Tomcat 太胖太慢了。

_演进：_ Java 阵营现在利用 **GraalVM 编译器** 进行 **AOT（提前编译）**，在打包时把所有反射和配置全部计算好，直接编译成去除了 JVM 和 Tomcat 的**纯二进制原生镜像（Native Image）**，让 Spring Boot 3 或 Quarkus 的启动时间缩短到**几十毫秒**。

**云原生是“轻功流派”：** 底层利用操作系统的 **Epoll 机制（非阻塞 I/O）**。一个线程就可以管几万个网络连接。请求进来了，丢给后台异步处理，线程立刻去接下一个请求。这使得它们在云端可以用极低的内存，顶住恐怖的并发海啸。
