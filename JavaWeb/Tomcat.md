Tomcat 靠堆砌操作系统线程来顶并发

Tomcat 的潜意识里有几个假设：
- 资源是无限且独占的： 物理机常年开着，启动慢一点（花 5 秒、10 秒）没关系，因为一年到头也重启不了几次。
- 内存可以尽情挥霍： 物理机有的是内存，所以我启动时疯狂扫描所有的 Jar 包、预先创建几百个线程（每个线程吃 1MB 内存）常驻在内存里，我觉得理所当然。

# Tomcat 线程模型的局限（重）

## BIO模式 Thread-per-request（一个请求一个操作系统内核线程）模型。

当客户端（浏览器）向 Tomcat 发送连接请求时，Tomcat 会从线程池中分配一个线程伺候这个连接，负责 读取网络数据 -> 解析HTTP协议 -> 执行Servlet代码（业务逻辑、查数据库）-> 组装响应 -> 写回网络数据

Java 的线程在 JVM 底层是 1:1 映射到操作系统内核线程上的，有几个并发请求就会占用几个操作系统内核线程。

## 现代的 NIO 模式

NonBlocking IO： 解决的是网络等待和业务计算解耦

现代的 Tomcat 引入了多路复用技术，将网络连接和真正的业务处理解耦

负责处理分配连接的 Acceptor & Poller 线程，是极少数的内核线程，利用操作系统的epoll机制，管住成千上万的网络连接。如果连接只是干挂着（keep-alive）不发数据，不需要占用专门的内核线程。

Worker线程（工作线程池）：当某个连接真正有HTTP请求数据传送过来，Poller线程才会从线程池中抓 Worker 线程出来，把请求丢给它执行 Servlet。

执行完毕立刻回收，Servlet执行到数据返回给客户端了，Worker线程立刻回到池子中，伺候下一个连接，不需要等待连接断开。

## 虚拟线程（Virtual Threads）

解决的是代码遇到阻塞的IO操作的时候，JVM 会直接把这个虚拟线程从内核线程上“卸载”下来，让宝贵的内核线程，挂在另一个就绪的虚拟线程，直接别的业务，等待数据库的数据返回了，JVM接收到通知，将刚才放进内存歇着的虚拟线程，重新“挂载”到任意一个空闲的内核线程上，继续往下执行内核代码。

一个请求对应一个 Java 虚拟线程，如果这个虚拟线程被阻塞，会自动让出内核线程，让内核线程去执行别的请求。

## 在高并发或者 AI Agent 这种超长耗时的阻塞请求下，线程池会迅速耗尽，内存飙升。

演进： 现代架构改用 **Netty（异步非阻塞）**、**Go 语言协程（Goroutine）**，利用单线程事件循环（Epoll 机制）或极轻量的协程（只占几 KB），用极低的资源消耗顶住十几万连接。

**冷启动阻碍了 Serverless（慢）：** 传统 Tomcat 启动时要高频扫描类路径、通过反射加载 Bean，耗时数秒。而 Serverless 追求极致的按量付费、瞬间拉起、用完即毁。Tomcat 太胖太慢了。

_演进：_ Java 阵营现在利用 GraalVM 编译器 进行 AOT（提前编译），在打包时把所有反射和配置全部计算好，直接编译成去除了 JVM 和 Tomcat 的纯二进制原生镜像（Native Image），让 Spring Boot 3 或 Quarkus 的启动时间缩短到几十毫秒。

云原生是“轻功流派”： 底层利用操作系统的 **Epoll 机制（非阻塞 I/O）**。一个线程就可以管几万个网络连接。请求进来了，丢给后台异步处理，线程立刻去接下一个请求。这使得它们在云端可以用极低的内存，顶住恐怖的并发海啸。
