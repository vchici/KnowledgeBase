# 字符 & 字段

字段：在数据库的表格中，字段就是“列”，char(10)或者varchar(10)就是在定义这个字段的类型和长度。

字符：不是字节！一个字符占多少个字节，取决于数据库使用的字符集（Encoding）

- **在 ASCII 编码下：**
	- 1 个英文字母 = 1 个字符 = 1 个字节。
- **在 UTF-8 编码下（最常用）：** 
	- 1 个英文字母/数字 = 1 个字符 = 1 个字节。
	- 1 个常用汉字 = 1 个字符 = **3 个字节**。
- **在 UTF8mb4 编码下（支持表情包）：**
	- 1 个 Emoji 表情（如 😊）= 1 个字符 = **4 个字节**。

# char & varchar

char：固定长度的字符类型，定义一个char(10)字段，不管实际存储的字符长度，只会占用是个字符的空间，少于十个字符，剩余部分用空格填充。

varchar：可变长度的字符类型，理论上容纳65535个字符。

# blob & text

blob 用于存储二进制数据，比如图片、音频、视频、文件等；但实际开发中，我们都会把这些文件存储到 OSS 或者文件服务器上，然后在数据库中存储文件的 URL。

## 存储加密密钥、证书或二进制 Token

某些加密算法生成的私钥、数字证书（如 SSL 证书）、或者特定格式的二进制 Token，它们本身就是一串无规律的二进制字节流，不适合用文本（String）表示。这时用 `BLOB`（或 `VARBINARY`）存储最安全、最高效。

## 游戏服务器中的“玩家镜像数据”（Player State）

一个玩家身上可能有几百种属性、装备、任务进度、背包道具。如果把这些全部拆成数据库的表和字段，每次上下线都要进行几十次 `JOIN` 查询，数据库会直接崩溃。

游戏服务器在内存中把玩家的所有数据序列化（Serialize）成一个二进制结构体（或者 Protocol Buffers 格式），然后直接当成一个 `BLOB` 塞进数据库。

读写只有一条记录，速度极快。

## session（会话）持久化 

当用户登录系统后，他的登录状态、购物车信息等会话数据（Session），有些框架会选择序列化后加密，直接以二进制的形式存在数据库的 BLOB 字段中，用于实现分布式登录。

## 本地/轻量级数据库（如 SQLite）

在一些不需要对接云端 OSS 的独立客户端软件、嵌入式设备或者手机 App（使用 SQLite 数据库）中，因为没有网络或者为了离线可用，用户的头像或小图标依然会直接用 BLOB 存放在本地数据库里。

text 用于存储文本数据，比如文章、评论、日志等。

短文本首选 `varchar`，但是在面对长度不固定且可能非常巨大的纯文本内容，选择 `TEXT`

## 存储富文本内容（HTML / Markdown）

这是 `TEXT` 最常见的岗位。比如博客文章、新闻内容、商品详情页的描述、论坛的帖子等。

- 这些内容不仅字数多，里面还包含了大量的 HTML 标签、样式代码或者 Markdown 语法。
    
- **为什么不用 VARCHAR？** 因为 `VARCHAR` 最大限制通常是 65535 字节（在 UTF-8 编码下大约只能存 2 万多个汉字）。一篇长文加上各种前端排版代码，很容易就会爆表。而 `LONGTEXT` 可以存储高达 4GB 的文本。

## 存储大段的日志或审计记录（Log / Audit Trail）

在一些系统内部，需要记录某些业务操作的详细上下文，或者保存支付接口返回的原始报文（Response XML / JSON）。

- 这些数据通常只用于后台排查错误或审计，不需要用来做高频的条件筛选。
    
- 它们体积大且长度极不固定，非常适合直接塞进一个 `TEXT` 字段里扔在后台。

## 存储非结构化的 JSON 配置或配置快照

虽然现在的关系型数据库（如 MySQL 5.7+、PostgreSQL）都原生支持了更强大的 `JSON` 类型，但在一些老系统、或者对 JSON 检索要求不高的场景下，大家依然习惯用 `TEXT` 来存一串序列化后的 JSON 字符串。

- **例如：** 用户的自定义个性化设置快照、某个审批流的动态节点配置、电商订单在下单那一刻的商品快照（防止商家后续修改商品信息导致纠纷）。

## 存储大模型的 Prompt（提示词）或 AI 对话上下文

在 2020 年代 AI 应用爆发后，`TEXT` 迎来了一个巨大的新舞台：

- **Prompt 模板：** 复杂的 Agent 提示词可能长达几千字，包含各种 Few-shot 示例。
    
- **对话历史（Chat History）：** 用户的多轮对话上下文、或者 RAG（检索增强生成）召回回来的大段参考文档，都需要用 `TEXT` 类型持久化到数据库中。

## 技术进阶：同样是存文本，它和 VARCHAR 有什么本质区别？

很多新手会问：“既然 `VARCHAR(20000)` 也能存不少字，我为什么非要用 `TEXT`？” 这涉及到数据库底层的**行溢出机制（Row Overflow）**：

- **VARCHAR 是“行内存储”：** 数据库读取一行数据时，会把这一行所有的 `VARCHAR` 内容一起读进内存。如果你一行里定义了多个很大的 `VARCHAR`，即便你没填满，它也会占用宝贵的行数据页（Page）空间，导致数据库一页能存的记录变少，降低查询效率。
    
- **TEXT 是“行外存储”：** 数据库在行内只存一个 **10 字节左右的指针**。真正的文本内容被藏在另外一个专门的数据页里。当你 `SELECT *` 没查这个 TEXT 字段时，数据库根本不会去读那段大文本，**完全不占用行内存**。只有当你明确要查这个字段时，它才会顺着指针去把文本捞出来。

## 使用 TEXT 类型的“避坑指南”

- **绝对不要对 TEXT 字段进行 `ORDER BY` 或 `GROUP BY`：** 因为 TEXT 太大，数据库无法在内存中对它进行排序，会直接触发**磁盘临时表**，性能会慢得像乌龟爬。如果非要排序，建议对文章的“发布时间”或“ID”排序。
    
- **尽量避免 `SELECT *`：** 养成好习惯，需要什么字段就查什么。如果不小心把不必要的 `TEXT` 字段一起查出来，会消耗大量的网络带宽和数据库 I/O。
    
- **无法建立全文本普通索引：** 你不能直接给一个 `TEXT` 字段加 `idx_content` 这样的普通索引。如果你需要给文章内容做关键词模糊搜索，正确的做法是搞一个 **Elasticsearch（ES）**，或者使用数据库的 `FULLTEXT` 全文索引（但性能有限）。

# DATETIME 和 TIMESTAMP 有什么区别？

TIMESTAMP 时区敏感

DATETIME 时区无感

假设现在有一张表，里面有两个字段：`dt_col (DATETIME)` 和 `ts_col (TIMESTAMP)`。 你在 **东八区（北京时间）** 往里插入了一条数据：`2026-06-12 12:00:00`。

当不同的用户去读取这条数据时：

- **北京用户（东八区）** 看到的结果：
	- `DATETIME`: `2026-06-12 12:00:00`
	- `TIMESTAMP`: `2026-06-12 12:00:00`
- **东京用户（东九区，比北京快 1 小时）** 看到的结果：
	- `DATETIME`: `2026-06-12 12:00:00` （毫无变化，依然是字面值）
	- `TIMESTAMP`: **`2026-06-12 13:00:00`** （数据库自动根据东京时区加了 1 小时！）

**底层的逻辑是：** > * `DATETIME` 就像你在本子上写下一行字，不管谁来看，字面意思都一样。

`TIMESTAMP` 在写入时，MySQL 会把你当前时区的时间转成**格林威治标准时间（UTC）**存进底层；当你读取时，MySQL 又会把 UTC 时间转成你**当前连接的时区**显示出来。

## 致命的隐患：2038年问题 (Y2K38)

由于 `TIMESTAMP` 只有 4 个字节，它能表示的最大秒数是 $2^{31}-1$（因为是有符号整型）。这就导致它能表示的最大时间只能到 **2038 年 1 月 19 日 03:14:07**。

如果你的系统在 2038 年之后还在运行，使用 `TIMESTAMP` 的字段将会直接**溢出崩溃**。虽然现在距离 2038 年还有十多年，但对于需要存储长期合同、婴儿出生日期、房产年限等长远数据的业务，现在就绝对不能用 `TIMESTAMP` 了。

## 实际开发中怎么选择？（避坑指南）

过去很多老教程推荐使用 `TIMESTAMP`，因为觉得它省空间（4字节 vs 8字节，注：老版本DATETIME占8字节）。但在硬件成本低廉的今天，这点空间差距可以忽略不计。

目前大厂的业界主流规范（如阿里开发手册）普遍建议使用 `DATETIME`。

- **安全长寿：** 完美避开 2038 年灾难，支持到 9999 年，适合做长周期业务。
    
- **直观可控：** 存进去是什么，读出来就是什么，排查问题（比如对账、查日志）时不会因为服务器、数据库、客户端的时区配置混乱而产生莫名其妙的 Bug。
    
- **性能几乎无差：** 现代 MySQL（5.6.4+）优化了 `DATETIME` 存储，只需要 5 个字节，与 TIMESTAMP 的 4 字节相比微乎其微。

# 记录货币

如果是电商、交易、账单等涉及货币的场景，建议使用 DECIMAL 类型，因为 DECIMAL 类型是精确数值类型，不会出现浮点数计算误差。

例如，`DECIMAL(19,4)` 可以存储最多 19 位数字，其中 4 位是小数。

```SQL
CREATE TABLE orders (
    id INT AUTO_INCREMENT,
    amount DECIMAL(19,4),
    PRIMARY KEY (id)
);
```

如果是银行，涉及到支付的场景，建议使用 BIGINT 类型。可以将货币金额乘以一个固定因子，比如 100，表示以“分”为单位，然后存储为 `BIGINT`。这种方式既避免了浮点数问题，同时也提供了不错的性能。但在展示的时候需要除以相应的因子。

## 既然 `DECIMAL` 已经能做到精准，为什么在处理超高并发、海量数据的银行核心支付系统中，技术专家往往更倾向于推荐 `BIGINT`（以“分”或“厘”为单位）呢？

### 性能的终极压榨（CPU 与计算速度）

- **BIGINT：** 是数据库和 CPU 的**原生整数类型**。加减乘除运算直接由 CPU 的硬件加法器/乘法器执行，速度是纳秒级别的。
    
- **DECIMAL：** 是**定制的定点数类型**。因为要保证绝对精准，数据库底层通常是用**类似字符串或变长字节数组**的形式来存储的。当你在 SQL 中对 `DECIMAL` 进行加减乘除时，CPU 无法直接计算，而是需要通过软件层面（数据库引擎的算法）将它拆解、转换、逐位计算。
    
> **结果：** 在海量批处理、利息计算、日终对账等需要处理数亿级数据的银行场景下，`BIGINT` 的计算速度通常比 `DECIMAL` **快数倍甚至一个数量级**。

### 存储空间的精简与索引效率

在银行系统里，数据量是天文数字，存储空间的节省意味着更少的磁盘 I/O 和更高的内存缓存（Buffer Pool）命中率。

- **BIGINT：** 固定占用 **8 个字节**，可以表示高达 $\pm 9 \times 10^{18}$ 的数字（即便是以“分”为单位，也能存下千亿个“小目标”）。
    
- **DECIMAL(19,4)：** 它的存储通常是按组划分的（例如在 MySQL InnoDB 中，每 9 位数字占用 4 字节，小数点后的 4 位单独占用 2 字节），大约需要 **9 个字节** 甚至更多。
    
更重要的是，**`BIGINT` 作为索引的效率远高于 `DECIMAL`**。在进行范围查询、求和（`SUM`）或 `GROUP BY` 时，更小的字节和原生的整数比较能极大地提升数据库的吞吐量。

### 分布式架构与跨语言通信

现代银行系统大量采用微服务和分布式架构（如 Java、Go、C++ 等多语言混合）。

- 当使用 `BIGINT` 时，在所有编程语言中它都对应基础的 `long` 或 `int64` 类型，数据在网络传输（RPC、JSON 序列化）时**完全不会失真**，解析速度极快。
    
- 当使用 `DECIMAL` 时，Java 需要映射为 `BigDecimal`，Go 需要用专门的 `shopspring/decimal` 库。如果团队中有人不小心在某个微服务里把 `DECIMAL` 反序列化成了 `double`，精度瞬间就会崩溃。**使用 `BIGINT` 从源头上抹平了跨系统、跨语言传递的风险。**

### 银行特有的业务习惯

银行对账的本质是“数硬币”。 在银行的业务逻辑中，货币的最小流通单位通常就是确定的（比如人民币是“分”，甚至内部记账到“厘”或“毫”）。直接用整数（1元 = 100分）来思考和存储，非常符合资金对账的直觉——**所有的资金变动，本质上都是整数个最小货币单位的转移。**


## 为什么不推荐使用 FLOAT 或 DOUBLE？

因为 FLOAT 和 DOUBLE 都是浮点数类型，会存在精度问题。

在许多编程语言中，`0.1 + 0.2` 的结果会是类似 `0.30000000000000004` 的值，而不是预期的 `0.3`。
