# Binlog

> 归档日志，记录所有**已提交**的写操作，属于 MySQL Server 层（与存储引擎无关）。

## 核心特征

| 维度 | 说明 |
|------|------|
| 所属层级 | Server 层，所有存储引擎共用 |
| 记录内容 | 逻辑日志（SQL 语句或行变更） |
| 写入时机 | 事务提交后写入，先写 page cache 再 fsync 落盘 |
| 用途 | 主从复制、数据恢复、CDC 数据同步 |
| 开启方式 | 默认关闭，`my.cnf` 中 `log-bin=mysql-bin` 开启 |

## 三种写入格式

```ini
# my.cnf 配置
binlog_format = STATEMENT | ROW | MIXED
```

| 格式 | 记录内容 | 优点 | 缺点 |
|------|---------|------|------|
| `STATEMENT` | 原始 SQL 语句 | 日志量小 | 函数（`NOW()`、`UUID()`）在从库重放结果不一致 |
| `ROW` | 每行数据的变更前后镜像 | 精确，无歧义 | 日志量大（批量操作会产生大量行记录） |
| `MIXED` | 自动选择，一般用 STATEMENT，遇不安全函数切 ROW | 兼顾大小与精度 | 判断逻辑复杂 |

MySQL 5.7+ 默认 `ROW`。

## 写入机制（两阶段提交）

Binlog 与 [Redolog](#Redolog) 通过**两阶段提交**保证一致性，避免主从数据错乱：

```
1. 写 Redolog（prepare 状态）   ── 存储引擎层
2. 写 Binlog                   ── Server 层
3. 写 Redolog（commit 状态）    ── 存储引擎层
```

崩溃恢复时的判断：
- 若 Binlog 已写入 → 提交事务（认为事务已成功）
- 若 Binlog 未写入 → 回滚事务（认为事务未成功）

这样从库通过 Binlog 重放时，能与主库状态严格一致。

## 典型应用

### 主从复制

主库把 Binlog 发给从库，从库重放 SQL/行变更，实现数据同步。

```
主库（写操作）→ Binlog → IO Thread → Relay Log → SQL Thread → 从库数据
```

### 数据恢复

通过 `mysqlbinlog` 工具解析并重放指定时间段的日志：

```bash
# 查看某段时间的变更
mysqlbinlog --start-datetime="2025-01-01 00:00:00" \
            --stop-datetime="2025-01-01 12:00:00" \
            mysql-bin.000123 | mysql -u root -p

# 按位置恢复
mysqlbinlog --start-position=154 --stop-position=1024 mysql-bin.000123 | mysql -u root -p
```

### CDC 数据同步（缓存一致性）

Canal 伪装成 MySQL 从库，实时解析 Binlog 得知表行变更，用于异步更新 Redis 缓存，详见 [数据更新](Redis/数据更新.md#L40)。

```
MySQL 写入 → 产生 Binlog → Canal 解析 → DEL Redis / 写入 MQ
```

## 与 Redolog 的区别

| 对比项 | Binlog | Redolog |
|--------|--------|---------|
| 层级 | Server 层 | InnoDB 存储引擎层 |
| 类型 | 逻辑日志 | 物理日志（页的修改） |
| 写入方式 | 追加写，文件切换 | 循环写，固定大小 |
| 作用 | 复制、恢复、CDC | 崩溃恢复（crash-safe） |
| 必需性 | 可选（默认关） | 必需 |

# Redolog

> 重做日志，InnoDB 引擎层的物理日志，记录数据页的修改，保证 crash-safe（崩溃恢复）。

## 核心特征

| 维度 | 说明 |
|------|------|
| 所属层级 | InnoDB 存储引擎层（非 Server 层） |
| 记录内容 | 物理日志（"哪个数据页的哪个偏移改成了什么"） |
| 写入时机 | 事务执行过程中持续写入，先写日志再改数据页（WAL） |
| 作用 | 崩溃恢复，保证已提交事务不丢失 |
| 必需性 | InnoDB 必需，不可关闭 |

## WAL 机制（Write-Ahead Logging）

Redolog 的核心思想：**先写日志，再写数据页**。

```
传统方式（无 WAL）：              WAL 方式：
修改数据页 → 落盘                 写 Redolog → 落盘
                                  修改数据页（内存）→ 异步落盘
```

**为什么这样能提升性能？**
- 数据页是随机写（页分散在磁盘各处），慢
- Redolog 是顺序写（追加到日志文件尾部），快
- 把"随机写"转化为"顺序写"，事务提交只需等 Redolog 落盘

**为什么能保证不丢数据？**
- 崩溃时数据页可能没落盘，但 Redolog 已落盘
- 重启后用 Redolog 重放，把未落盘的数据页恢复出来

## 循环写结构

Redolog 不是无限追加的（这点与 Binlog 不同），而是固定大小、循环写入：

```
                                              write pos（下一条将写在这里）
                                              ↓
┌──────────────────┬──────────────────────────┬──────────────────┐
│      已刷盘       │        未刷盘            │      空闲         │
│     可覆盖        │     （待恢复用）          │     未写入        │
└──────────────────┴──────────────────────────┴──────────────────┘
                   ↑
                   checkpoint（已刷盘边界，之前可覆盖）

write pos ──→ 不断前移（写日志）
checkpoint ──→ 跟在后面追（刷盘后前移）
```

位置关系说明：
- `checkpoint` 指向"已刷盘"与"未刷盘"的交界，它**之前**是已刷盘（可覆盖）
- `write pos` 指向"未刷盘"与"空闲"的交界，它**之前**是已写日志（已刷盘 + 未刷盘），**之后**是空闲
- 所以"未刷盘"区域 = `checkpoint` 到 `write pos` 之间，这段日志已写但数据页未落盘

三个区域的含义：

| 区域 | 位置 | 含义 | 能否覆盖 |
|------|------|------|---------|
| 已刷盘 | checkpoint 之前 | 日志已写，且对应的数据页已落盘 | ✅ 可覆盖（恢复时不需要） |
| 未刷盘 | checkpoint → write pos 之间 | 日志已写，但数据页还在内存（Buffer Pool），未落盘 | ❌ 不可覆盖（崩溃恢复要重放） |
| 空闲 | write pos 之后 | 还没写日志 | ✅ 可写入新日志 |

**为什么"未刷盘"区域是崩溃恢复的关键？**
- Redolog 的作用就是保证"数据页没落盘也不丢数据"
- 崩溃时，"未刷盘"区域对应的内存数据页丢失了
- 重启后重放这段 Redolog，把数据页重新恢复出来

**write pos 与 checkpoint 的运动关系**：
- 两者**同向移动**，write pos 在前写新日志，checkpoint 在后追赶
- 当 write pos 绕一圈快追上 checkpoint 时（空闲区域耗尽，日志写满），必须停下推进 checkpoint，强制把数据页刷盘

```ini
# my.cnf 配置 Redolog 大小
innodb_log_file_size = 1G       # 单个文件大小
innodb_log_files_in_group = 2   # 文件个数，共 2G
```

Redolog 写满会触发"强行刷盘"，导致业务停顿，所以大事务场景需要调大 `innodb_log_file_size`。

## 刷盘策略（innodb_flush_log_at_trx_commit）

控制 Redolog 何时从 page cache 真正写入磁盘：

| 值 | 行为 | 安全性 | 性能 |
|----|------|--------|------|
| `0` | 每秒由后台线程刷盘 | 最低（崩溃丢 1 秒数据） | 最高 |
| `1` | 每次事务提交都刷盘 | 最高（不丢数据） | 最低 |
| `2` | 每次提交写 page cache，每秒刷盘 | 中（OS 崩溃才丢） | 中 |

```ini
innodb_flush_log_at_trx_commit = 1   # 默认值，生产推荐
```

**双 1 配置**：Redolog 的 `innodb_flush_log_at_trx_commit=1` + Binlog 的 `sync_binlog=1`，是 MySQL 数据安全的黄金标准。

两个"1"分别控制两个日志的刷盘时机：

| 参数 | 作用对象 | =1 的含义 | 保证 |
|------|---------|----------|------|
| `innodb_flush_log_at_trx_commit` | Redolog（引擎层） | 每次事务提交都把 Redolog 刷到磁盘 | 主库崩溃后能恢复 |
| `sync_binlog` | Binlog（Server 层） | 每次事务提交都把 Binlog 刷到磁盘 | 从库能正确同步 |

**为什么必须"双"1 才安全？** 因为 Redolog 和 Binlog 通过[两阶段提交](#写入机制两阶段提交)协作，单独开一个不够：

- 只开 Redolog 的 1：主库能恢复，但 Binlog 没落盘 → 从库丢数据 → 主从不一致
- 只开 Binlog 的 1：从库能同步，但 Redolog 没落盘 → 主库丢数据 → 主从不一致
- 双 1：两个日志都落盘，主库可恢复 + 从库可同步，主从严格一致

**性能代价**：双 1 是最安全但最慢的配置，每次事务提交都要两次 fsync。高并发场景可适当降级（如 `=2` + `sync_binlog=100`），但金融交易场景必须坚持双 1。

降级配置的含义：

| 配置 | 原值（双1） | 降级值 | 区别 | 崩溃可能丢失 |
|------|------------|--------|------|-------------|
| `innodb_flush_log_at_trx_commit` | `1` 每次提交 fsync | `2` 每次提交只写 page cache，每秒 fsync | 不等 fsync，只写内存 | OS 崩溃丢 1 秒 |
| `sync_binlog` | `1` 每次提交 fsync | `100` 每 100 次提交才 fsync 一次 | 攒 100 个事务再落盘 | 最多丢 100 个事务 |

**降级为什么能提速？** fsync 涉及磁盘机械寻道，非常慢。双 1 每次提交等 2 次 fsync；降级后几乎不等 fsync（只写内存），性能可提升数倍到数十倍。

**降级的风险**：
- MySQL 进程崩溃（被 kill）：page cache 仍在 OS 内核，重启不丢数据
- OS 崩溃/断电：page cache 丢失，RedoLog 丢 1 秒、Binlog 丢 100 个事务 → 主从不一致

## 崩溃恢复（crash-safe）

重启时 InnoDB 的恢复流程：

```
1. 读取 Redolog，找到 checkpoint 位置
2. 重放 checkpoint 之后的 Redolog（已提交 + 未提交的事务都重放）
3. 用 Undolog 回滚未提交的事务
4. 数据恢复到崩溃前已提交的状态
```

与 [Binlog](#Binlog) 两阶段提交的配合：
- Redolog 处于 `prepare` 状态时崩溃 → 查 Binlog
- Binlog 已写入 → commit（认为事务成功）
- Binlog 未写入 → rollback（配合 [Undolog](#Undolog) 回滚）

## 与 Binlog 的关键区别（补充）

Binlog 章节已有对比表，这里补充几个易混淆点：

| 对比项 | Redolog | Binlog |
|--------|---------|--------|
| 日志类型 | 物理日志（页级修改） | 逻辑日志（SQL/行变更） |
| 写入粒度 | 事务执行中持续写 | 事务提交时一次性写 |
| 文件结构 | 固定大小循环写 | 无限追加，按大小切换文件 |
| 幂等性 | 重放需配合 checkpoint | 可重复重放（逻辑操作幂等） |

**为什么需要两份日志？**
- Redolog 是引擎层的，保证崩溃恢复，但只在单机有效
- Binlog 是 Server 层的，用于主从复制，但写入时机晚于事务执行
- 两者职责不同，缺一不可，通过两阶段提交协调

# Undolog

> 回滚日志，InnoDB 引擎层的逻辑日志，记录数据修改前的旧值，用于事务回滚和 MVCC。

## 核心特征

| 维度 | 说明 |
|------|------|
| 所属层级 | InnoDB 存储引擎层 |
| 记录内容 | 逻辑日志（修改前的反向操作） |
| 写入时机 | 修改数据页之前先写 Undolog |
| 作用 | 事务回滚、MVCC 多版本读 |
| 生命周期 | 事务提交后不立即删除，等没有读事务依赖时才清理 |

## 记录内容（反向操作）

Undolog 记录的是"如何撤销这次修改"，与 [Redolog](#Redolog) 的"如何重做这次修改"正好相反：

| 操作 | Undolog 记录的反向操作 |
|------|----------------------|
| `INSERT` | `DELETE`（删除刚插入的行） |
| `DELETE` | `INSERT`（恢复被删除的行） |
| `UPDATE` | `UPDATE`（改回旧值） |

```
执行 UPDATE user SET age=30 WHERE id=1：

Undolog：  记录 age 的旧值（如 20）→ 用于回滚或 MVCC 读旧版本
Redolog：  记录"数据页 X 偏移 Y 改成 30" → 用于崩溃恢复
数据页：   age 实际改成 30
```

## 两大核心作用

### 1. 事务回滚

事务执行失败或被 `ROLLBACK` 时，用 Undolog 反向操作恢复数据：

```sql
BEGIN;
UPDATE account SET balance = balance - 100 WHERE id = 1;  -- 写 Undolog（旧值 balance=500）
UPDATE account SET balance = balance + 100 WHERE id = 2;  -- 写 Undolog（旧值 balance=200）
-- 发生异常
ROLLBACK;
-- InnoDB 读取 Undolog：
--   把 id=1 的 balance 改回 500
--   把 id=2 的 balance 改回 200
```

**与 Redolog 的协作**：崩溃恢复时，Redolog 重放所有事务（包括未提交的），再用 Undolog 回滚未提交的事务，保证"已提交的保留，未提交的回滚"。

### 2. MVCC 多版本并发控制

这是 Undolog 最容易被忽略但极重要的作用——**为读操作提供历史版本**。

```
事务 A（未提交）：UPDATE user SET age=30 WHERE id=1;  -- age 旧值 20 存入 Undolog

事务 B（读）：SELECT age FROM user WHERE id=1;
  → 当前数据页 age=30（未提交，不可见）
  → 沿 Undolog 链找到旧版本 age=20（可见）
  → 返回 20
```

每行数据的旧版本通过 Undolog 串成**版本链**：

```
当前数据(age=30) → Undolog v2(age=20) → Undolog v1(age=18) → NULL
                   ↑                    ↑
              Read View 判断可见性
```

**RC vs RR 隔离级别的区别**就在于 Read View 的生成时机：
- **RC（读已提交）**：每条 SELECT 都生成新的 Read View → 总能读到最新已提交版本
- **RR（可重复读）**：事务开始时生成一次 Read View → 整个事务读到的版本固定

## 生命周期与 purge 线程

Undolog **不是事务提交就删除**，因为可能还有其他事务在读它的旧版本：

```
事务 A 提交后：
  ├── 没有其他事务依赖该 Undolog → purge 线程清理
  └── 事务 B 正在读旧版本 → 保留，等 B 结束后清理
```

```ini
# my.cnf 配置 Undolog
innodb_undo_directory = ./        # Undolog 文件目录
innodb_undo_logs = 128            # 回滚段数量
innodb_max_undo_log_size = 1G     # 单个 Undolog 文件最大大小
innodb_undo_log_truncate = ON     # 自动回收空间
```

**长事务的危害**：长事务不结束，它产生的 Undolog 无法被 purge 清理，导致：
- Undo 表空间膨胀
- 历史版本链变长，降低查询性能
- 占用大量内存

## 三种日志的协作关系

```
                    ┌─────────────────────────────────┐
                    │         事务执行流程              │
                    └─────────────────────────────────┘
修改数据前          修改数据时          提交时           崩溃恢复
    │                  │                │                │
    ▼                  ▼                ▼                ▼
写 Undolog        写 Redolog        写 Binlog       重放 Redolog
(旧值，用于回滚)   (物理修改，       (逻辑变更，      回滚未提交事务
                   用于崩溃恢复)     用于复制)        (用 Undolog)
                                        │
                                        ▼
                                   两阶段提交
                                   (Redolog + Binlog)
```

| 日志 | 层级 | 类型 | 写入时机 | 核心作用 |
|------|------|------|---------|---------|
| [Binlog](#Binlog) | Server 层 | 逻辑日志 | 事务提交时 | 复制、恢复、CDC |
| [Redolog](#Redolog) | 引擎层 | 物理日志 | 事务执行中 | 崩溃恢复 |
| Undolog | 引擎层 | 逻辑日志 | 修改数据前 | 回滚、MVCC |

**一句话总结**：
- Redolog 保证"已提交的不丢"
- Undolog 保证"未提交的可回滚" + "读不到未提交的"
- Binlog 保证"主从数据一致"
