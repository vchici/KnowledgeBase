# in和exists的区别？

我们假设有两张表：外表 `A`（如：用户表）和内表 `B`（如：订单表）。

`IN` 的逻辑：先查内表，再查外表

`IN` 会先执行子查询，把内表 `B` 中所有满足条件的记录查出来，缓存在内存中。然后遍历外表 `A`，看 `A` 的字段在不在这个缓存的集合里。

```python
# IN 的伪代码
B_list = select_all_from_B() # 先把子查询查完，结果存在内存里
for a in A:
    if a.id in B_list:      # 在内存中比对
        print(a)
```

`EXISTS` 的逻辑：先查外表，再循环查内表

`EXISTS` 会先遍历外表 `A`，每遍历到 `A` 的一条记录，就会把这条记录的关联字段代入到子查询中去执行一次。如果子查询有返回结果（哪怕只有一条），`EXISTS` 就返回 `True`，当前外表记录保留。

```python
# EXISTS 的伪代码
for a in A:
    # 每次循环都去执行一次子查询，一旦找到一条满足条件的 B，立刻终止本次子查询
    if has_any_match_in_B(a.id): 
        print(a)
```

## 外表大，内表小 —— 首选 `IN`

- **规则：** 小表驱动大表。
    
- **原因：** 如果内表 `B` 只有 100 条数据，`IN` 先把这 100 条数据捞出来缓存在内存里，外表 `A` 有 100 万条数据，就可以直接走外表 `A` 的索引去内存里匹配，效率极高。

## 外表小，内表大 —— 首选 `EXISTS`

- **规则：** `EXISTS` 命中索引更关键。
    
- **原因：** 如果外表 `A` 只有 100 条数据，内表 `B` 有 100 万条。`EXISTS` 只需要循环 100 次。只要内表 `B` 的关联字段建立了索引，这 100 次循环每次都是极其快速的索引精准命中，不需要把内表 100 万条数据加载到内存。

## 致命陷阱：对 `NULL` 值的处理完全不同

这是编写 SQL 时最容易踩坑的地方，特别是当它们前面加上 `NOT`（即 `NOT IN` 和 `NOT EXISTS`）的时候。

### 当子查询的结果集包含 `NULL` 时：

- **`IN` 的崩溃表现：** 如果子查询的结果集里包含一个 `NULL` 值（例如：`[1, 2, NULL]`），那么 `NOT IN` 的整个查询将**直接返回空结果集（什么都查不到）**！
    
    - _原因：_ 在 SQL 中，任何值与 `NULL` 进行等值比较（如 `id != NULL`）的结果都是 `Unknown`（未知），导致整条判断直接失效。
        
- **`EXISTS` 的稳定表现：** `EXISTS` 的子查询里不管是有 `NULL` 还是有具体的值，它只关心“有没有行返回”。即便子查询返回的是一条包含了 `NULL` 的记录，它也算“有数据返回”，从而返回 `True`。因此 `NOT EXISTS` 不受 `NULL` 值的影响。
    
> 💡 **避坑铁律：** 如果你无法保证子查询的列里 100% 没有 `NULL` 值，**一律使用 `NOT EXISTS` 代替 `NOT IN`**。

### 语法与书写习惯

- **`IN` 的书写：** 子查询只返回单列。
    
    ```SQL
    SELECT * FROM User 
    WHERE id IN (SELECT user_id FROM Orders WHERE price > 100);
    ```
    
- **`EXISTS` 的书写：** 子查询通常用 `SELECT 1`，且必须包含外表和内表的**关联条件**。
    
    ```SQL
    SELECT * FROM User u 
    WHERE EXISTS (
        SELECT 1 FROM Orders o 
        WHERE o.user_id = u.id AND o.price > 100 -- 必须有关联条件
    );
    ```
    
    _(注：子查询里写 `SELECT 1`、`SELECT *` 或 `SELECT id` 在性能上没有任何区别，因为 EXISTS 只看有没有行，不看具体列的值，业界习惯写 `1`。)_

# 外表和内表

把数据库的两表关联，想象成代码里的 **双重 `for` 循环**

```python
# 这是一个双重 for 循环示例
for x in Table_A:          # 外层循环：Table_A 就是“外表”
    for y in Table_B:      # 内层循环：Table_B 就是“内表”
        if x.id == y.id:
            print(match)
```

**外表（Outer Table / 驱动表）：** 放在**外层循环**的表。数据库会先读取外表，然后一条一条地吐出数据，去触发内层的查询。

**内表（Inner Table / 被驱动表）：** 放在**内层循环**的表。外表每吐出一条数据，内表就要被动地去“匹配”一次。

很多新手会想：“反正最后都要两两匹配，谁在外面、谁在里面，总的匹配次数不都是 $A \times B$ 吗？为什么会有先后顺序的影响？”

两者的巨大差距，不在于“循环了多少次”，而在于“内层循环能不能走索引”！

### 场面一：外表 100 条，内表 100 万条（且内表建了索引）

数据库去外表捞数据，外表在外面，一共只循环 **100 次**。

每一次循环，拿着外表的 ID 去内表（100万条）里找。

**关键来了：** 因为内表有索引（比如 B+ 树），在 100 万条数据里找一个 ID，只需要 3~4 次磁盘 I/O（极快）。

**总消耗：** 100 次外层循环 $\times$ 极快的索引查找 = **瞬间完成**。

### 场面二：外表 100 万条，内表 100 条（且内表建了索引）

数据库去外表捞数据，外表在外面，要循环 **100 万次**！
    
每一次循环，拿着外表的 ID 去内表（100条）里找。
    
虽然内表很小，找起来也快。
    
**总消耗：** **100 万次**外层循环的 CPU 开销和上下文切换 + 100 万次内层查找 = **明显变慢，甚至卡死**。

**核心结论：** 先查哪个表（谁当驱动表），决定了哪张表可以享受**“被索引精准降维打击”**的福利。被频繁反复查找的那张表（内表），必须得有索引。

## 小表驱动大表

写 SQL 或者数据库优化时，应该让“数据量小（或者过滤后结果集小）的表”作为外表（驱动表），让“数据量大（但建立了索引）的表”作为内表（被驱动表）。

其核心目的，是为了**减少外层循环的次数**。

小表指的不是`SELECT COUNT(*)` 结果小的表，**“小表”指的是“经过 WHERE 条件过滤后，参与实际循环的结果集更小的表**。

比如：`A` 表有 100 万条数据，`B` 表有 10 万条数据。看起来 `B` 是小表对吧？ 但是你的 SQL 语句是：

```SQL
SELECT * FROM A JOIN B ON A.id = B.a_id WHERE A.status = '今天刚注册';
```

如果“今天刚注册”的用户在 `A` 表里只有 50 条，那么此时 `A` 过滤后的结果集（50条）远小于 `B` 表（10万条）。这时候，MySQL 的优化器就会聪明地选择 **`A` 表作为驱动表（外表）**。

### 为什么外表有索引也阻止不了全表扫描

假设外表（驱动表）是 `User`（1万条数据），它的 `id` 字段是有索引的。 当它作为外表去驱动内表时，数据库要做的第一步是：**把这 1 万条数据一条一条地拿出来。**

请问：**既然要把 1 万条数据全都拿出来，索引还能帮上忙吗？**
- **帮不上大忙。** 索引的本质是加速“精准查找（Where id = 5）”**或**“范围查找（Where id > 100）”。
- 如果没有任何过滤条件，数据库必须把外表的所有行都遍历一遍。此时，外表有没有索引，它都得老老实实地去磁盘或内存里做一次**全表扫描**（或者全索引扫描），把 1 万条数据吐出来。

所以，外表是否有索引，无法改变它要“循环 1万次”的宿命。

### 如果“内外表都有索引”，会发生什么？（名场面）

如果外表和内表都有索引，那这就是最完美的理想状态。但两者的分工截然不同：

#### 外表的索引：负责“瘦身”（减少外层循环次数）

如果 SQL 语句带有 `WHERE` 过滤条件，外表的索引就派上大用场了：

```SQL
SELECT * FROM User u JOIN Order o ON u.id = o.user_id 
WHERE u.age = 18; -- 外表过滤条件
```

- 如果 `User.age` 有索引，数据库就会通过索引**精准抓出**所有 18 岁的用户。
- 假设 1 万个用户里只有 50 个是 18 岁，那么通过外表索引的过滤，**外层循环次数直接从 1 万次暴跌到 50 次！**
- **外表索引的作用：** 帮助外表快速完成“瘦身”，让驱动表变得极小。

#### 内表的索引：负责“拦截”（实现降维打击）

外表“瘦身”后，剩下 50 条数据。接下来，这 50 条数据会轮流进入内层循环：

```SQL
-- 伪代码：内层循环在干的事情
SELECT * FROM Order WHERE user_id = [当前外表用户的id];
```

这就是为什么大家都在强调**内表索引**的原因。内表要是没有索引，外表的每条数据都会触发一次内表的全表扫描，算法复杂度直接恶化为 $O(N \times M)$。

- **外表的索引是用来“过滤数据”的**。只有当你写了 `WHERE 外表字段 = xxx` 时，外表的索引才有用，用来减少循环总次数。
    
- **内表的索引是用来“等值匹配”的**。它是为了防止内表每次都进行全表扫描。不管你写不写 `WHERE`，只要是两表关联（`ON a.id = b.a_id`），内表的索引就必须发挥作用。

## 如果两个表不是等值匹配

两表关联的条件可以是大于（`>`）、小于（`<`）、不等于（`!=`）、`BETWEEN...AND` 甚至是 `LIKE`。

### 范围匹配（`BETWEEN...AND` / `>` / `<`）

有一张`订单表（Orders）`记录了每一笔交易的时间，另一张`促销活动表（Campaigns）`记录了每个活动开始和结束的时间。现在你想知道**每笔订单分别是在哪个促销活动期间内派生的**。

```SQL
SELECT o.order_id, o.order_time, c.campaign_name
FROM Orders o
JOIN Campaigns c 
  ON o.order_time BETWEEN c.start_time AND c.end_time; -- 非等值匹配
```

如果 `Orders` 是外表，`Campaigns` 是内表：

- 数据库遍历外表 `Orders`，吐出一笔订单，时间是 `2026-06-12 12:00:00`。
    
- 进入内表 `Campaigns` 查找满足 `start_time <= '2026-06-12 12:00:00' AND end_time >= '2026-06-12 12:00:00'` 的记录。
    
- **内表的索引还有用吗？** **有用！** 如果内表的 `start_time` 建立了 B+ 树索引，数据库可以通过索引快速定位到第一个满足条件的活动，而不需要全表扫描。但是，由于是范围查找，它不能像等值匹配那样“瞬间精准抓取一条”，而是需要顺着 B+ 树的叶子节点链表进行一段**范围扫描（Range Scan）**。

### 不等值匹配（`!=` 或 `<>`）

比如你有一张 `Staff（员工表）`，你想找出**所有不在同一个部门、但薪资完全相同的员工组合**，用来做薪酬公平性分析。

```SQL
SELECT e1.name, e2.name, e1.salary
FROM Staff e1
JOIN Staff e2 
  ON e1.salary = e2.salary AND e1.department_id != e2.department_id; -- 包含不等值匹配
```

如果条件里纯粹只有 `!=`（比如 `ON a.name != b.name`）：

- **索引直接失效！** B+ 树索引的强项是告诉你“谁在哪里”，但无法高效地告诉你“谁不等于谁”。
    
- 如果内表只有 `!=` 条件，那么外表每吐出一条数据，内表都必须做一次**全表扫描**来找出所有不相等的人。这会引发前面提到的 $O(N \times M)$ 灾难。

当两表关联不是等值匹配时，MySQL 等数据库在选择执行策略时会非常纠结：

1. **无法使用 Hash Join（哈希连接）：** 现代数据库（如 MySQL 8.0、PostgreSQL）引入了极其高效的 `Hash Join` 算法。如果两表是**等值匹配**，数据库会把小表在内存中建一个哈希表，大表一扫，瞬间就能匹配出来，连内表索引都不需要。 **但是，Hash Join 只能用于等值匹配。** 一旦你用了 `>`、`<`、`!=`，数据库就无法计算哈希值，这个暴风级别的优化手段直接失效。
    
2. **退化为 Block Nested Loop（块嵌套循环）或全表扫描：** 如果碰上非等值匹配，且没有合适的范围索引，MySQL 只能采用最原始的办法：把外表的一堆数据读到内存缓冲区（Join Buffer），然后全表扫描内表，在内存里一条条去比对是否满足 `BETWEEN` 或 `!=`。

## 总结：非等值匹配的避坑指南

1. **能用等值尽量用等值：** 非等值连接的性能通常明显弱于等值连接，在写复杂 SQL 时，尽量看看能不能通过增加辅助字段，把非等值转换成等值。
    
2. **范围关联一定要加索引：** 像 `ON a.time BETWEEN b.start AND b.end` 这种业务，**内表的范围字段（如 `start`）必须建索引**，否则大表关联时系统会瞬间卡死。

##  问题一：如果大表没索引，小表有索引，该怎么驱动？

既然我们前面推导出了“内表必须有索引，否则会触发大面积全表扫描”的结论，那么：

- 如果让**有索引的小表**作为内表（被驱动表）。
- 让**无索引的大表**作为外表（驱动表）。

这样听起来，大表哪怕有 10万 条，一条条吐出来，去内表（小表，比如 50 条）里找，因为小表有索引，找得也很快。总计算量大约是 $10\text{万} \times 3 \sim 4 \text{次} \approx 40\text{万次}$。这比让无索引的大表当内表（导致计算量暴增到 $50 \times 10\text{万} = 500\text{万次}$）要好得多！

**所以，在这种极端情况下，确实应该让“有索引的小表”作为内表。**

### 性能的“两难困境”与现代数据库的解法

但是，让大表在外面循环 10 万次，CPU 切换和数据读取的开销依然很大。如果两表是**等值匹配**（比如 `ON 大表.id = 小表.id`），现代数据库（MySQL 8.0+）根本不会在这两个烂方案里二选一，它会直接掏出大招：**Hash Join（哈希连接）**。

1. **它完全不管索引了。**
2. 数据库会把**小表**（50条）加载到内存里，在内存中构建一个**哈希表（Hash Table）**。
3. 然后对**大表**（10万条）进行**一次性全表扫描**，每读出一条数据，就在内存里算一下哈希值，看能不能对上小表。
4. **总计算量：** 扫描一次大表（10万次） + 内存哈希比对。速度极快，直接绕过了“谁当内表都没索引”的尴尬局面。

## ## 问题二：内外表的选择是靠人写 SQL 决定，还是靠 MySQL 自动优化？

答案是：**绝大多数情况下，靠 MySQL 的优化器（Optimizer）自动决定。**

在早期的数据库或者某些特定数据库（如 Oracle 的 RBO 模式）中，表的顺序确实由你写的 SQL 顺序决定（比如写在 `FROM` 后面的顺序，或者 `LEFT JOIN` 的左右顺序）。

但是在现代 MySQL 中，采用的是 **CBO（基于成本的优化器，Cost-Based Optimizer）**。

- 不管你在 SQL 里把哪个表写在前面，也不管你用的是 `JOIN`、`LEFT JOIN` 还是 `RIGHT JOIN`（在不破坏业务逻辑的前提下）。
- MySQL 在真正执行前，都会通过**计算各种方案的 CPU 和磁盘 I/O 成本**，自动选择那张“过滤后结果集最小的表”去驱动大表。

## 问题三：既然 MySQL 自动优化，为什么这还是个重要的知识点？

既然机器帮我们做了，为什么面试还要考？为什么架构师还要懂？原因有以下三个致命场景，MySQL 优化器也会“翻车”：

### 1. 数据库的“统计信息”可能失真（最常见原因）

MySQL 决定谁是大表、谁是小表，不是每次执行 SQL 都去 `COUNT(*)` 查一遍（那太慢了），而是根据数据库自己维护的**统计信息（Statistics）**来估算的。 如果这张表最近发生过大规模的增删改，或者统计信息过期了，MySQL 可能会**误把大表当成小表**，从而做出错误的执行计划。这时候就需要程序员懂这个原理，去手动执行 `ANALYZE TABLE` 更新统计信息，甚至用 `STRAIGHT_JOIN` 强行指定驱动顺序。

### 2. 复杂的复合索引，优化器会“猜错”

当你的 `WHERE` 条件非常复杂，涉及多个表的多个字段，且存在多个联合索引时，MySQL 计算成本的算法可能会陷入局部最优，导致选错驱动表。 只有你懂“小表驱动大表”和“内表索引”的原理，你才能看懂 `EXPLAIN`（执行计划）里的 `rows` 和 `Extra`，从而知道 MySQL 是不是“变傻了”，进而去优化你的索引设计。

### 3. 指导你如何建立“联合索引”

知道“内表需要索引，外表需要过滤索引”的原理，能直接决定你的索引建在哪张表的哪个字段上。

- **比如：** 如果通过原理判断出 `A` 表过滤后一定是小表，`B` 表是大表。那你立刻就能反应过来：`B` 表的关联字段上必须建索引！而不用盲目地给两张表的所有字段都加上索引，浪费磁盘空间和写入性能。

## 问题四：为什么说即使小表大表都没有索引，还是应该小表做外表

### 核心影响：Join Buffer 的空间利用与磁盘 I/O

在没有索引时，MySQL 会采用 **Block Nested Loop（块嵌套循环）** 算法。它绝对不会傻到每读一条外表数据，就去磁盘扫描一遍内表。它的真实操作是：

1. **小表驱动大表（$A \to B$）：**
    
    MySQL 拨了一块内存叫 `Join Buffer`。它把**小表 $A$ 的全部 100 条数据一次性塞进内存缓冲区**。
    
    然后，它对大表 $B$ 进行**一次全表扫描**。大表 $B$ 的每条数据进内存，跟缓冲区里的 100 条数据批量比对。
    
    - **结果：** 大表 $B$ **只被扫描了 1 次**！磁盘 I/O 极低。
        
2. **大表驱动小表（$B \to A$）：**
    
    MySQL 试图把大表 $B$ 塞进 `Join Buffer`。但是缓冲区太小了（默认几百 KB），只能塞下大表 $B$ 的前 1 万条数据。
    
    接下来，扫描一次小表 $A$（100条）来跟这 1 万条比对。
    
    缓冲区满了，清空；再塞入大表 $B$ 的第 2 万条数据……再扫描一次小表 $A$。
    
    - **结果：** 为了把大表分批装完，内表（小表 $A$）**被迫重复进行了很多次全表扫描**，产生了大量的磁盘 I/O 交换。
        
> **结论：** 即使没有索引，**小表驱动大表**也能保证驱动表尽可能一次性全部装入内存，从而**极大地减少内表的磁盘扫描次数**。

### 在 `ON` 关联条件上，外表的索引“形同虚设”

假设有 SQL 如下，且 `User.id` 和 `Order.user_id` 都有索引：

```SQL
SELECT * FROM User u JOIN Order o ON u.id = o.user_id;
```

如果 MySQL 选了 `User` 作为外表，那么执行流程是：

1. **外表（User）：** MySQL 需要把 `User` 表里的数据一条一条全部捞出来（全表扫描）。既然是**全表扫描**，那么 `User.id` 上的索引就没有任何意义了。因为索引是为了帮你“少读数据”，而你现在要读“所有数据”。
    
2. **内表（Order）：** 外表每吐出一个 `id`（比如 `5`），内表就要执行一次 `WHERE user_id = 5`。这时候内表的索引被疯狂调用。
    
**在这个 `ON` 的阶段，外表的索引确实没有任何用武之地，形同失效。**

### 为什么说外表的索引没有真正失效？（WHERE 阶段的降维打击）

但是，实际开发中我们几乎不会写不带 `WHERE` 条件的大表盲联。只要有 `WHERE` 条件，外表的索引就是**整条 SQL 性能的生死线**。

再看这个例子：

```SQL
SELECT * FROM User u JOIN Order o ON u.id = o.user_id
WHERE u.status = 'active' AND u.age > 30; -- 外表的过滤条件
```

如果 `User` 依然是外表：

1. **外表（User）利用索引“自救”：** MySQL 绝对不会去全表扫描 `User`。它会立刻启用 `User.status` 或 `User.age` 上的索引，把符合条件的人精准抓出来。
    
2. 假设 `User` 表原本有 100 万行，通过外表自身的索引过滤后，只剩下了 100 行。
    
3. **决定性的影响出现了：** 外层循环的次数从 **100 万次暴跌到了 100 次**！

如果此时外表没有索引，它就得全表扫描 100 万行去找出这 100 个人，那整条 SQL 在第一步（外表阶段）就已经卡死了。

## 今日份干货总结

- 写 SQL 的时候，**不用刻意去调换 `JOIN` 前后的表顺序**来迎合“小表驱动大表”，现代 MySQL 会帮你调。
    
- 你需要做的是：通过 `WHERE` 条件尽量把驱动表“过滤”得越小越好，并且**确保被驱动的表（内表）的关联字段上有高效的索引**。

**我们作为开发者，唯一要努力做到的，就是通过 `WHERE` 条件把某一张表的数据过滤到最小，然后确保另一张被关联的表上，关联字段有索引。** 剩下的，现代数据库的优化器比我们想象的要聪明得多。

# SQL查询语句的执行顺序

先执行 FROM 确定主表，再执行 JOIN 连接，然后 WHERE 进行过滤，接着 GROUP BY 进行分组，HAVING 过滤聚合结果，SELECT 选择最终列，ORDER BY 排序，最后 LIMIT 限制返回行数。

WHERE 先执行是为了减少数据量，HAVING 只能过滤聚合数据，ORDER BY 必须在 SELECT 之后排序最终结果，LIMIT 最后执行以减少数据传输。

| 执行顺序 | SQL 关键字  | 作用                          |
| ---- | -------- | --------------------------- |
| ①    | FROM     | 确定主表，准备数据                   |
| ②    | ON       | 连接多个表的条件                    |
| ③    | JOIN     | 执行 INNER JOIN / LEFT JOIN 等 |
| ④    | WHERE    | 过滤行数据（提高效率）                 |
| ⑤    | GROUP BY | 进行分组                        |
| ⑥    | HAVING   | 过滤聚合后的数据                    |
| ⑦    | SELECT   | 选择最终返回的列                    |
| ⑧    | DISTINCT | 进行去重                        |
| ⑨    | ORDER BY | 对最终结果排序                     |
| ⑩    | LIMIT    | 限制返回行数                      |

这也是为什么有时候在 SELECT 子句中定义的别名不能在 WHERE 子句中使用得原因，因为 WHERE 是在 SELECT 之前执行的。

# MySQL目录下的可执行文件

MySQL 的 bin 目录下有很多可执行文件，主要用于管理 MySQL 服务器、数据库、表、数据等。比如说：

- mysql：用于连接 MySQL 服务器
- mysqldump：用于数据库备份，对数据备份、迁移或恢复时非常有用
- mysqladmin：用来执行一些管理操作，比如说创建数据库、删除数据库、查看 MySQL 服务器的状态等。
- mysqlcheck：用于检查、修复、分析和优化数据库表，对数据库的维护和性能优化非常有用。
- mysqlimport：用于从文本文件中导入数据到数据库表中，适合批量数据导入。
- mysqlshow：用于显示 MySQL 数据库服务器中的数据库、表、列等信息。
- mysqlbinlog：用于查看 MySQL 二进制日志文件的内容，可以用于恢复数据、查看数据变更等。
