# 问题现象

Redis和MySQL中都不存在查询的key值

该情况同时大量发生，MySQL承受高负载查询

# 解决方案

## 前端过滤不合法的查询key值

无法拦截合规但不存在的key

## 缓存中存 <该key值，null>

浪费内存（如果黑客生成很多随机不存在key值），适合数据量小的情况

潜在数据不一致（如果key值之后被创建了，缓存的对象没过期，这个可以通过插入数据时主动清除/更新缓存）

 [持久化](持久化.md)

## 布隆过滤器

适合数据量大的情况（亿级数据量）

bit数组存储key值被多个不同hash函数计算得到的结果

比如，计算结果为26，则在数组26下标处标记为1

### 不会误判不存在，但会误判存在

在该key值被多个不同hash函数计算得到结果中，bit数组中如果有不为1的情况，则该key值一定不存在

但是即使bit数组中都为1的情况，该key值也有可能不存在

### 降低误判率

可以通过哈希函数数量和数组大小降低误判率，设置在1%左右，在 1% 的误判率下，平均存储 1 条数据，大约需要 9.6 个比特位（bit），预估数据量在100万条时，需要的比特位数量约956万个bit，换算成物理内存约1.14MB

### bit数组实现

哈希函数的结果，如何对应到具体的bit位上？

哈希函数的结果是一个固定长度的整数，例如一个32位的int整数，范围在 $-2147483648 \sim 2147483647$ 之间

整数 % bit 数组总长度 = 数组中 bit 的位置 X

计算机中能操作的最小内存单位是字节，如果通过 long (64bits) 和 int (32bits) 组成的普通数组模拟，还需要对 X 进一步用64或32取模，得到对应数组中的下标，然后用64或32取余，得到对应变量内部的第几个bit
