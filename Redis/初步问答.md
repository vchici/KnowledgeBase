## 🗺️ Redis 整体学习结构（全景图）

一个完整的 Redis 知识体系通常包含以下四个层级：

```
[第1层：基础数据结构] ➡️ [第2层：高级特性与应用] ➡️ [第3层：高可用与集群] ➡️ [第4层：底层原理与调优]
```

## 🚀 速成程序员：核心死磕重点

时间有限的情况下，请把 80% 的精力放在以下通关重点上：

### 1. 五大基础数据结构与“名门绝技” 🔥

不要只背命令，要记**应用场景**。面试和写代码最常问的就是“什么场景用什么结构”。

- **String（字符串）**：常规计数、Session 共享、分布式锁。
    
- **Hash（哈希）**：存储对象信息（如用户信息、商品信息），比 String 更省空间。
    
- **List（列表）**：简单消息队列、朋友圈点赞列表（关注时间线）。
    
- **Set（集合）**：利用交集、并集做“共同关注”、“好友推荐”或去重。
    
- **ZSet（有序集合）**：🔥 **必考！** 各种排行榜（热榜、积分榜）、带权重的任务队列。
    

### 2. 缓存三驾马车（面试重灾区） 🔥🔥🔥

这三个概念在面试中命中率几乎是 100%，不仅要懂概念，更要背过**解决方案**。

- **缓存穿透**：故意访问不存在的数据。
    
    - _速成解法_：缓存空对象、**布隆过滤器（Bloom Filter）**。
        
- **缓存击穿**：单个热点 Key 突然过期，海量请求直击数据库。
    
    - _速成解法_：热点数据永不过期、加**分布式锁**。
        
- **缓存雪崩**：大量 Key 同时过期，或者 Redis 宕机。
    
    - _速成解法_：过期时间加随机随机值、搭建高可用集群、限流降级。
        

### 3. Redis 为什么这么快？ 🔥

高频理论题，记住以下三点组合拳：

1. **纯内存操作**（基础）。
    
2. **核心是单线程模型**（避免了多线程的上下文切换和锁竞争，注意：Redis 6.0+ 引入多线程只是为了处理网络IO，执行命令依然是单线程）。
    
3. **采用了 I/O 多路复用机制**（如 Linux 的 `epoll`），让单线程也能高效处理海量连接。
    

### 4. 分布式锁的实现 🔥

分布式系统必用，速成阶段不需要手写，但必须懂原理：

- **基础版**：`SET key value NX PX 30000`（原子性释放需要用 Lua 脚本）。
    
- **进阶版**：直接看开源神库 **Redisson** 的原理（看懂看门狗 `Watchdog` 自动续期机制）。
    

### 5. 持久化与高可用（懂概念与优缺点即可）

不用去背配置文件，但面试官喜欢问对比：

- **RDB（快照）**：定时把内存全量存盘。优点：恢复快；缺点：容易丢最后一次的数据。
    
- **AOF（日志）**：来一条命令写一条日志。优点：数据安全；缺点：文件大、恢复慢。
    
- _速成结论_：生产环境通常采用 **AOF+RDB 混合持久化**。
    
- **集群方案**：分清**哨兵模式**（自动主从切换，保证高可用）和 **Redis Cluster**（多主多从，横向扩容，解决单机内存瓶颈）。

# 回答

兄弟，你抓到了核心！**“什么场景用什么结构”**这句话，确实就是在讲 Redis 的**具体实战用法**。

在传统关系型数据库（如 MySQL）里，我们不管存什么，基本都是建表、定义字段、写 SQL。但在 Redis 里，它没有表的概念，它是一个 **键值对（Key-Value）** 数据库。Value 的部分有多种“形状”（也就是数据结构），比如字符串、哈希表、列表。

开发时的核心思考转变为：**“我当前的这个业务需求（场景），用 Redis 的哪种数据结构实现，性能最高、最省内存、写代码最优雅？”**

下面我直接用 **Java (Spring Boot + RedisTemplate)** 的真实代码，带你看看最常问的 4 个实战场景到底是怎么玩的。

## 🛠️ Redis 实战代码与场景演练

### 场景一：用户登录后的 Token 存取（使用 String 结构）

- **为什么用 String？** 因为一个 Token 对应一个用户的 JSON 字符串，直接整存整取，最简单高效。
    

Java

```
@Autowired
private StringRedisTemplate redisTemplate;

// 1. 用户登录成功，生成Token并存入Redis，设置1小时过期
public void saveUserToken(String token, UserInfo user) {
    String key = "auth:token:" + token;
    String userJson = JSONObject.toJSONString(user); 
    
    // String 结构的对应操作是 opsForValue()
    redisTemplate.opsForValue().set(key, userJson, 1, TimeUnit.HOURS);
}

// 2. 拦截器校验登录状态：根据Token获取用户信息
public UserInfo getUserInfo(String token) {
    String key = "auth:token:" + token;
    String userJson = redisTemplate.opsForValue().get(key);
    
    if (userJson == null) {
        return null; // Token过期或不存在
    }
    return JSONObject.parseObject(userJson, UserInfo.class);
}
```

### 场景二：商品详情页的商品信息（使用 Hash 结构）

- **为什么用 Hash？** 如果用 String 存商品，每次修改库存或价格，都要把整个大 JSON 捞出来修改再塞回去。而 Hash 就像一个 Map，可以**单独修改商品下的某一个属性（Field）**，非常灵活且省流量。
    

Java

```
// 1. 缓存商品详情
public void cacheProduct(String productId, Map<String, String> productInfo) {
    String key = "product:" + productId;
    
    // Hash 结构的对应操作是 opsForHash()
    // 假设 productInfo 包含 {"name":"iPhone 15", "price":"5999", "stock":"100"}
    redisTemplate.opsForHash().putAll(key, productInfo);
}

// 2. 用户下单，扣减单个属性（库存）
public void deductStock(String productId, int quantity) {
    String key = "product:" + productId;
    
    // 关键：直接利用 Redis 针对 Hash 的原子操作，把库存直接减去 quantity
    redisTemplate.opsForHash().increment(key, "stock", -quantity);
}
```

### 场景三：朋友圈点赞列表 / 社交时间线（使用 List 结构）

- **为什么用 List？** List 是一个双向链表，它的特点是**有序、元素可重复**。朋友圈点赞或最新消息，要求按“时间先后顺序”排列，用 List 往头部一直塞（LPUSH）就能完美实现。
    

Java

```
// 1. 某人给小明的朋友圈(post:99)点赞
public void likePost(String postId, String userId) {
    String key = "post:likes:" + postId;
    
    // List 结构的对应操作是 opsForList()
    // LPUSH：把最新的点赞用户塞到列表最前面
    redisTemplate.opsForList().leftPush(key, userId);
}

// 2. 分页查看前10个点赞的人
public List<String> getTopLikes(String postId) {
    String key = "post:likes:" + postId;
    
    // LRANGE：获取下标 0 到 9 的数据（前10条记录）
    return redisTemplate.opsForList().range(key, 0, 9);
}
```

### 场景四：今日热歌排行榜（使用 ZSet 结构）

- **为什么用 ZSet？** ZSet（有序集合）每个元素都会关联一个 `Score`（分数），Redis 会自动按照分数帮你排好序。这在传统数据库里需要写复杂的 `ORDER BY`，在 Redis 里是天然支持的。
    

Java

```
// 1. 每播放一次歌曲，该歌曲的分数（播放量）+1
public void playSong(String songId) {
    String key = "rank:hot_songs";
    
    // ZSet 结构的对应操作是 opsForZSet()
    // ZINCRBY：给指定歌曲的分数加 1
    redisTemplate.opsForZSet().incrementScore(key, songId, 1);
}

// 2. 获取播放量前3名的歌曲（降序排列）
public Set<String> getTop3Songs() {
    String key = "rank:hot_songs";
    
    // ZREVRANGE：获取分数从大到小前3名的歌曲ID
    return redisTemplate.opsForZSet().reverseRange(key, 0, 2);
}
```

# 回答

兄弟，**完全正确！** 你的理解已经非常精准了，抓住了这两个结构最核心的本质。

为了让你在面试或者跟人聊技术时显得更专业，我们可以用更贴合 Redis 官方的术语来复盘强化一下：

### 1. 关于 Hash 结构 ➡️ **完全正确**

- **你的理解**：存储的是 Map，由属性名和属性值组成。
    
- **技术规范补充**：在 Redis 官方术语里，这个 Map 的“属性名”叫做 **Field（字段）**，“属性值”叫做 **Value（值）**。
    
- **核心优势**：你可以像在 Java 里操作 `HashMap` 一样，直接通过 `HSET` / `HGET` 或者代码里的 `opsForHash()` 单独去读写、修改里面的某一个 Field，而不需要把整个大对象搬来搬去。
    

### 2. 关于 ZSet 结构 ➡️ **完全正确**

- **你的理解**：存储的是集合，可以修改元素的分数，Redis 会自动按分数排序。
    
- **技术规范补充**：在 Redis 官方术语里，ZSet（有序集合）里的“元素”叫做 **Member（成员）**，每个 Member 背后绑定的“分数”叫做 **Score（分值）**。
    
- **核心优势**：
    
    1. **Member（元素）是唯一的、不能重复的**（这符合“集合”的特征）。
        
    2. **Score（分数）是可以重复的，也是随时可以修改的**。
        
    3. **天然有序**：正如你所说，只要你修改了某个元素的分数（比如通过 `incrementScore` 让它的播放量加 1），Redis 内部会立刻利用跳表（SkipList）这种底层结构，**在内存中自动重新排序**。你不需要写任何排序算法，拿出来的时候直接就是排好序的队列。

# 回答

兄弟，学得快不是本事，用得稳、在生产环境不踩坑才是硬道理。面试官最喜欢问的不是“怎么用”，而是“**你在用这五大结构时，踩过什么坑？怎么优化的？**”

针对这五大基础数据结构，我为你整理了生产环境最致命的 **4 大核心考点与闭坑指南**。

## 🚨 考点一：BigKey（大 Key）问题 —— 毁掉 Redis 的头号杀手 🔥🔥🔥

这是 Redis 面试中**必问、必考、不考就等于没面过 Redis** 的绝对核心。

- **什么是 BigKey？**
    
    并不是指 Key 本身的字数多，而是指 **Value 占用的内存太大了**。
    
    - **String 结构**：单个 Value 大于 5KB（官方限制最大 512MB，但超过几KB就很危险了）。
        
    - **Hash/List/Set/ZSet 结构**：里面的元素个数超过 5000 个。
        
- **有什么危害？**
    
    Redis 是**单线程**执行命令的！如果一个大 Key 进出网络或者被删除（比如你执行 `DEL my_big_hash`），Redis 会卡死几秒钟去释放内存。这几秒内，其他所有线上请求全部阻塞，直接导致接口超时、数据库被拖垮，俗称“拒绝服务”。
    
- **怎么避坑/解决？**
    
    1. **拆分**：把一个大 Hash 拆成 10 个小 Hash（比如按用户ID取模 `user:1001:cart:1`, `user:1001:cart:2`）。
        
    2. **异步删除**：生产环境绝对**不要用 `DEL` 删除大 Key**！Java 代码中请使用 `redisTemplate.unlink(key)`。`UNLINK` 是非阻塞删除，它会把大 Key 的内存释放操作丢给后台线程异步处理，不会卡死主线程。
        

## 🚨 考点二：String 结构的隐形内存杀手（简单却耗内存） 🔥

- **业务场景**：假设你要存 1 亿个用户的登录状态，用户 ID 是 Long 型（8字节），状态是 0 或 1。
    
- **大坑**：如果你直接用 String 结构存，`set user:100001 1`，你以为只占了几个字节，结果存完 1 亿个发现内存爆了十几 GB！
    
- **为什么？**
    
    Redis 的 String 结构为了支持丰富的操作，底层有个叫 `RedisObject` 的结构体包裹着。不管你存多小的字，它额外的包装成本就至少有 24~32 字节。这就好比**买了一颗药丸，却用了一个大纸箱去打包送货**，空间利用率极低。
    
- **怎么避坑？**
    
    对于数量极其庞大、但数据很简单的场景：
    
    - 使用 **Bitmap（位图）**：它基于 String，但可以用“位(bit)”来存 0 或 1。1 个亿的用户只需要约 12MB 内存，直接节省千倍空间。
        
    - 使用 **Hash** 集中存储，利用 Redis 底层的 `ziplist/listpack` 紧凑编码。
        

## 🚨 考点三：List 结构的“消息积压”与“幽灵空转” 🔥

很多速成程序员喜欢用 List 结构的 `LPUSH` 和 `RPOP` 来做简单的消息队列。

- **大坑一：死循环空转（CPU 飙升）**
    
    如果用 `RPOP` 消费消息，代码通常需要写个 `while(true)`。如果队列空了，`RPOP` 会一直返回 `null`，你的代码就会疯狂死循环去死磕 Redis，导致 CPU 瞬间飙到 100%。
    
    - _解法_：代码中改用 **`BRPOP`（Block Right Pop）**。它是阻塞版的，如果队列没消息，它会线程挂起老老实实等在那里（可以设置超时时间），有消息了才唤醒，不浪费 CPU。
        
- **大坑二：消息丢失**
    
    `RPOP` 一旦把消息拿出来，Redis 内存里就删除了。如果你的 Java 业务代码这时候突然宕机，这条消息就彻底人间蒸发了。
    
    - _解法_：如果对消息安全性要求高，速成阶段请直接老老实实上 **RabbitMQ / RocketMQ**，不要用 List 硬撑。
        

## 🚨 考点四：ZSet 的性能陷阱与限流误区 🔥

ZSet 非常好用，支持范围查找（比如获取前10名，或者获取某个分数区间的元素）。

- **大坑：范围查询过大拖垮集群**
    
    ZSet 底层是跳表，支持 `ZRANGEBYSCORE` 这样的范围查询。但是，如果你的 ZSet 里面有几十万数据，你写代码时没注意，查了一个非常大的分数区间，导致一次性返回几万条数据。
    
    - **后果**：跟 BigKey 一样，单线程的 Redis 在网页传输几万条数据时直接卡死。
        
- **怎么避坑？**
    
    1. 严格控制 ZSet 的元素总量，定期清理老旧数据。
        
    2. 范围查询时，**必须带上 `LIMIT` 分页参数**（类似于 SQL 的 limit），每次只捞 10~50 条。
        

## 💡 一句话总结面试大招

> “我们在项目里用 **Hash** 存商品信息时，严格控制了 Field 的数量不超过 1000 个以防止 **BigKey** 阻塞单线程；同时在删除过期或作废的大型数据时，我们一律用 **`UNLINK`** 代替 `DEL` 实施异步清理，确保了线上 Redis 的高可用。”

兄弟，你问得非常细致，这两个问题都踩在了 Redis 底层设计和架构选型的核心点上。我们一个一个来掰扯清楚。

## 🙋‍♂️ 问题一：1亿用户存状态，怎么用“位图（Bitmap）”？

你这里有一个直觉上的小误区：**你把“用户 ID 的数据类型（Long型8字节）”和“用户 ID 的具体数值”混淆了。**

### 1. 位图的本质：它就是字符串！

首先回答你的疑问：**位图在 Redis 里，底层就是普通的 String（字符串）！** 它不是什么新发明的数据结构。

Redis 的字符串本质上是一个**二进制字节数组**（Byte Array）。既然是字节数组，它就是由无数个“位（bit，即 0 或 1）”排成一条线组成的。

### 2. 怎么解决你说的“有八位呢？”的疑问

我们根本不需要管 Long 型占几个字节，我们**直接把用户 ID 的“数字本身”，当成位图的“下标（Offset/偏移量）”**！

假设你的用户 ID 是 `100001`（这只是一个数字，不管它是用 Long 还是 Int 存的，它的值就是十万零一）。

- 在位图里，我们就去找**第 100001 个格子（bit）**。
    
- 如果用户在线，就把第 100001 个 bit 设置为 `1`。
    
- 如果用户离线，就把第 100001 个 bit 设置为 `0`。
    

### 3. 具体怎么写代码？

Redis 提供了专门操作 String 底层“位”的命令：`SETBIT` 和 `GETBIT`。在 Java 里长这样：

Java

```
// 假设我们的位图大Key名字叫 "user:online:status"
String bitmapKey = "user:online:status";
long userId = 100001L; // 你的 Long 型用户 ID

// 1. 设置用户 100001 的状态为 1（上线）
// opsForValue().setBit 的第二个参数就是偏移量，直接传用户ID！
redisTemplate.opsForValue().setBit(bitmapKey, userId, true);

// 2. 查询用户 100001 是否在线
Boolean isOnline = redisTemplate.opsForValue().getBit(bitmapKey, userId);
```

### 💡 为什么它能省下几千倍内存？

之前用 `set user:100001 1`，Redis 要为这“一个”用户创建一条独立的 Key-Value 记录，包括各种指针、过期时间、结构体包装，血亏 30 多字节。

而使用位图，**1 亿个用户，就是 1 亿个 bit（位）**。

$100,000,000 \text{ bits} \div 8 = 12,500,000 \text{ 字节 (Bytes)} \approx 12.5 \text{ MB}$。

也就是说，**你只需要一个 12.5MB 左右的普通 String Key，就能存下一亿用户的开关状态！** 这就是位图的恐怖之处。

## 🙋‍♂️ 问题二：用 List 做消息队列，到底是什么生产场景？

既然前面说了 List 做队列有“消息丢失”和“空转”的风险，为什么大家还要用它？因为在**轻量级、允许极小概率丢数据、图省事**的非核心业务场景下，它太香了。

如果为了一个简单的异步功能，去搭建、维护一套复杂的 RabbitMQ 或 Kafka 集群，不仅增加服务器成本，还增加了维护痛苦。

### 📊 真实的生产应用场景：**异步发送运营短信 / 邮件通知**

假设运营人员在后台点击了一个按钮：“给系统的 5 万个用户发送活动营销短信”。

- **如果是同步做**：后端代码要循环调用 5 万次第三方短信接口。这接口又慢，如果网页一直转圈，用户或运营早就崩溃了，甚至直接接口超时报错。
    
- **用 List 做轻量队列改进**：
    
    1. 运营点击发送，Java 后端立刻产生 5 万个“发短信任务”（包含手机号和内容）。
        
    2. 用 `LPUSH sms:queue 任务内容`，把这 5 万条数据疯狂塞进 Redis 的 List 里。
        
    3. 塞完立刻给前端返回：“系统已受理，正在后台异步发送。”（耗时仅需几毫秒，体验极佳）。
        
    4. 后台启动几个小微服务，用 `BRPOP sms:queue 30` 阻塞式地从 Redis 里一条条死磕拿任务，调用接口发短信。
        

### 为什么这个场景可以用 List 撑住？

1. **允许极低概率的丢失**：这只是营销短信，哪怕因为 Redis 宕机或网络闪断刚好丢了一两条，对核心业务（比如钱、订单）毫无影响，完全在业务容忍范围内。
    
2. **开发极快**：几行代码就搞定了，不需要配置复杂的交换机、队列和 MQ 监听器。
    

### 🚀 总结

- 如果是**涉及钱、订单、核心状态变更**的消息：**坚决不用 Redis List**，必须上正规 MQ。
    
- 如果是**发送通知、日志收集、应用内异步解耦、做任务流水**：用 Redis List 配合 `BRPOP` 是标准的“小快灵”速成方案。

### 1. 应对缓存穿透（Cache Penetration）的实战代码

- **体现点**：你不能只在 `if(user == null)` 时直接返回，你得**把空值塞回 Redis**，并且**过期时间要设置得很短**（比如 1 到 5 分钟），防止真的有人恶意刷接口。
    

Java

```
public UserInfo getUserWithPenetrationProtection(Long userId) {
    String cacheKey = "user:info:" + userId;
    
    // 1. 先查缓存
    String userJson = redisTemplate.opsForValue().get(cacheKey);
    
    // 🔥 亮点：如果是我们之前缓存的“空对象占位符”，直接返回 null，不查库
    if ("EMPTY_PLACEHOLDER".equals(userJson)) {
        return null; 
    }
    
    if (StrUtil.isNotEmpty(userJson)) {
        return JSONObject.parseObject(userJson, UserInfo.class);
    }
    
    // 2. 缓存没命中，查数据库
    UserInfo user = userMapper.selectById(userId);
    
    if (user == null) {
        // 🔥 亮点：数据库没有，缓存空对象，防止下次再穿透！过期时间设短一点（2分钟）
        redisTemplate.opsForValue().set(cacheKey, "EMPTY_PLACEHOLDER", 2, TimeUnit.MINUTES);
        return null;
    }
    
    // 正常数据缓存
    redisTemplate.opsForValue().set(cacheKey, JSONObject.toJSONString(user), 1, TimeUnit.HOURS);
    return user;
}
```

### 2. 应对缓存击穿（Cache Breakdown）的实战代码

- **体现点**：当热点 Key 过期时，**只能允许一个线程去查库**，其余线程在外面等着或者重试。这必须用到分布式锁（如 Redisson）。
    

Java

```
@Autowired
private RedissonClient redissonClient;

public ProductInfo getProductWithBreakdownProtection(Long productId) {
    String cacheKey = "product:detail:" + productId;
    String productJson = redisTemplate.opsForValue().get(cacheKey);
    
    if (StrUtil.isNotEmpty(productJson)) {
        return JSONObject.parseObject(productJson, ProductInfo.class);
    }
    
    // 🔥 亮点：缓存崩了，立刻加分布式锁，锁的 Key 要跟商品绑定
    String lockKey = "lock:product:" + productId;
    RLock lock = redissonClient.getLock(lockKey);
    
    try {
        // 尝试获取锁，最多等 3 秒，上锁后 10 秒自动释放
        if (lock.tryLock(3, 10, TimeUnit.SECONDS)) {
            // 双重检查（Double Check）：拿到锁后，再查一次缓存，可能前一个线程已经把数据查好放进去了
            productJson = redisTemplate.opsForValue().get(cacheKey);
            if (StrUtil.isNotEmpty(productJson)) {
                return JSONObject.parseObject(productJson, ProductInfo.class);
            }
            
            // 真正查库
            ProductInfo product = productMapper.selectById(productId);
            redisTemplate.opsForValue().set(cacheKey, JSONObject.toJSONString(product), 30, TimeUnit.MINUTES);
            return product;
        } else {
            // 没拿到锁的线程，证明有人在查库了，老老实实休眠一下然后重试
            Thread.sleep(100);
            return getProductWithBreakdownProtection(productId); // 递归重试
        }
    } catch (InterruptedException e) {
        return null;
    } finally {
        if (lock.isHeldByCurrentThread()) {
            lock.unlock(); // 释放锁
        }
    }
}
```

### 3. 应对缓存雪崩（Cache Avalanche）的实战代码

- **体现点**：绝对不能在代码里写死一个统一的过期时间（比如统一1小时），必须**在基础时间上加上一个随机的秒数**，让它们的过期时间错开。
    

Java

```
public void cacheBatchProducts(List<ProductInfo> list) {
    for (ProductInfo product : list) {
        String key = "product:detail:" + product.getId();
        
        // 🔥 亮点：基础过期时间 30 分钟 + 加上一个 1 到 5 分钟的随机数
        // 这样可以确保这批商品绝对不会在同一秒钟大面积集体过期
        int randomExpireTime = 30 + RandomUtil.randomInt(1, 5); 
        
        redisTemplate.opsForValue().set(key, JSONObject.toJSONString(product), randomExpireTime, TimeUnit.MINUTES);
    }
}
```
