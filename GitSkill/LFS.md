LargeFileStorage

# 影响

打破了 Git 分布式的管理，将大文件单独用中心化的方式管理

分布式：人人平等，每个人都拥有项目的所有历史记录（.git文件夹中），查看不同的历史版本无需网络

中心化：大文件单独存在远程LFS服务器中，本地.git文件夹中不包含所有历史版本，查看不同的历史版本需要网络重新下载

# 为什么要这样

普通的文本代码，Git可以做差量压缩，只保存不同版本之间的改动

大文件（视频、音频、图像、模型），无法做到差量压缩，只能全量保存

如果要保存所有的历史版本，大文件所占的体积是随着 修改次数 * 本身体积 而增加的

# 工作原理

Git 本身提供了一套极其强大的扩展机制，叫做 **Clean/Smudge 过滤器（Filter）**，Git LFS 正是寄生在这套机制上的

```text
【写入/提交文件 (Clean 阶段)】
真实大文件 (工作区) ---> [ LFS Clean Filter ] ---> 变成指针文件 (.git 暂存区)
                                |
                                v
                    (真实大文件被偷偷移入 .git/lfs/objects/)

----------------------------------------------------------------------

【读取/检出文件 (Smudge 阶段)】
指针文件 (.git 仓库) ---> [ LFS Smudge Filter ] ---> 变成真实大文件 (工作区)
                                ^
                                |
                    (LFS 根据指针哈希，从本地缓存或远程下载实体)

```

## 提交 （大文件 -> 指针）

把大文件 `tracker.mp4` 放入工作区并准备 `git add` 时：

1. LFS的 **Clean 过滤器** 拦截这个文件
2. LFS 计算这个大文件的 SHA-256 值，真实的大文件被复制到 `.git/lfs/objects/` 中
3. **Clean 过滤器** 生成一个几百字节的文本指针文件，交还给Git
4. Git 以为自己保存的是 `tracker.mp4` ，实际上只把几百字节的指针存进了版本历史

## 拉取/切分支 （指针 -> 大文件）

执行 `git clone`、`git pull` 或 `git checkout` 切换分支时：

1. Git从版本库中提取的是指针
2. LFS的 **Smudge过滤器** 拦截，并读取指针中的哈希值，去本地的 `.git/lfs/objects/` 寻找对应的真实大文件
3. 找到了，拿真实文件替换指针，没找到，LFS发起网络请求，去LFS服务器下载真实文件，放入工作区

## 具体流程

通过 `.gitattributes` 文件 告诉 Git，哪些文件需要 LFS

```bash
# 1. 开启 LFS 功能
git lfs install

# 2. 标记所有的 .psd 图片由 LFS 管理
git lfs track "*.psd"
```

此时，Git 会在项目根目录创建一个 `.gitattributes` 文件，内容类似于： `*.psd filter=lfs diff=lfs merge=lfs -text`

**这行规则就是最高指示**：告诉 Git 以后遇到 `.psd` 文件，必须调用 LFS 的 Clean/Smudge 过滤器。

推送时发生了两次网络传输：

```text
					+---> 1. 传输大文件实体 ---> LFS 存储服务器
                    |     (根据指针中的 SHA-256 路由)
git push origin main +
                    |
                    +---> 2. 传输代码与指针 ---> Git 代码服务器
```

LFS 优先运行，Git 随后传输，大文件上传失败，整个 `git push` 会直接报错终端，保证不会出现“代码传上去了，大文件丢了“的脱节情况。

拉取时协同作业：

Git客户端从代码服务器上下载最新代码和 `.gitattributes` 以及新指针

当 Git 尝试把指针写入工作区时，触发了 `.gitattributes` 里规定的 `filter=lfs`（Smudge 过滤器）

电脑自动连接 LFS 存储服务器，凭着指针里的 `oid` 认领对应的真实大文件

下载完成，文件在工作区复原
