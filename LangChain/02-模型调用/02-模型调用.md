左侧：大模型厂商提供的官方底层SDK，不同厂商SDK参数名和调用方法不同，每次调用都必须显式指定 `model`（模型名称）、`stream`（是否流式）、`temperature`（随机度）等所有工程参数。如果以后想把 DeepSeek 换成 Anthropic Claude，这段代码必须重写，因为不同厂商的 SDK 参数名和调用方法都不一样。

右侧：LangChain 统一接口方式，model已经是一个实例化好的 LangChain 模型对象。遵循 LangChain 的 `Runnable` 协议，统一了调用方法。无论底层是 OpenAI、DeepSeek 还是本地的 Llama，调用的方法**永远是 `invoke()` 或 `stream()`**。
模型名称、温度等参数在初始化 `model` 对象时就已经配置好了，调用时只需要关注“输入的内容”本身。

## 左侧有role content等复杂的结构，右侧传入“你是谁就可以”跑通

LangChain 在底层做了**自动类型转换（Auto-coercion）**

大模型底层（如左图）永远需要知道这段话是谁说的（即 `role: user`）。当你在 LangChain 的 `invoke()` 或 `stream()` 中直接传入一个纯字符串 `"你是谁？"` 时，LangChain 的底层接收器会启动一个预处理机制：

**“既然你只给了我一段文本，且没有指定角色，那这一定就是用户说的话。”**

LangChain 会在后台自动将这个字符串包装成一个标准的 **`HumanMessage`** 对象（等同于 `{"role": "user", "content": "你是谁？"}`）。

LangChain 的 `invoke` 接口设计非常包容，它支持三种输入层级，会自动在底层进行对齐：

- _高级（最简）_：`"字符串"` $\rightarrow$ 自动转为 `[HumanMessage(content="...")]`  
- _中级（图右上）_：`[{"role": "user", "content": "..."}]` $\rightarrow$ 自动转为对应的 Message 对象列表
- _低级（最明确）_：`[SystemMessage(...), HumanMessage(...)]` $\rightarrow$ 显式对象列表
