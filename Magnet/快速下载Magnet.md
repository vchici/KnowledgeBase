### 核心拆解：从 Magnet 转换为 Torrent 中间究竟发生了什么？

整个转换过程在底层其实是一个“海外服务器替你跨国寻宝”的过程，主要分为 3 个步骤：

```
[你贴入 Magnet 暗号] 
       │
       ▼
 1. 握手与广播 ──► [海外公网服务器] ──(高带宽/多线程)──► 全球 P2P/DHT 网络 (寻找活人)
                                                                 │
                                                                 ▼
 2. 抓取元数据 ◄────────────────(0.1秒瞬间下载说明书)─────────────┘
       │
       ▼
 3. 打包生成 ──► 注入初始优质 Tracker ──► 吐出 [.torrent 种子文件] 给你
```

1. **第一步：握手与广播（Handshake & Broadcast）** 你贴入的磁力链接只有一串 40 位的 Hash 字符（即特征码/暗号）。网站的服务器拿到这串暗号后，会立刻利用它自身的**公网 IP 和极高的网络权重**，向全球的 DHT 网络（分布式哈希表）和公共 Tracker 服务器大喊：“谁有这个特征码的文件？”
    
2. **第二步：抓取元数据（Metadata Retrieval）** 因为服务器在海外，网络几乎没有任何墙和内网限制，它能在 0.1 秒内瞬间联系到成百上千个正在做种的海外用户。一旦联系上，它不下载电影本体，而是**只下载这部电影的“说明书”（也就是元数据 Metadata）**。这份说明书里记录了电影的文件名、总大小、分块校验码等信息。
    
3. **第三步：打包与注入（Packaging & Injecting）** 网站后端将抓到的这份说明书（Metadata）格式化，并**自动往里面注入几条当前网络上最活跃、最优质的公共 Tracker 服务器地址**，最后打包成一个标准的 `.torrent` 文件，吐给你的浏览器。

### 转成 Torrent 之后，下载速度的“质变提升之处”在哪里？

正如你前面所说，种子和磁力底层的 P2P 传输速度极限是一样的，但转成 `.torrent` 注入客户端后，你会体验到**三个非常明显的速度提升和体验优化**：

#### 1. 零延迟起跑：省去了“卡 0%”的无谓等待

磁力链接像是个压缩包，你得先解压（获取种子）才能看里面有啥。而直接用 `.torrent`，**解压这一步已经在云端被网站替你做完了**。 你把种子丢进 qBittorrent 或 WebTorrent 的瞬间，进度条立刻从 0.1% 开始跳动速度，**直接进入数据下载阶段**。这对于追求“点开就看、无缝播放”的你来说，是体验上的绝对质变。

#### 2. “按顺序下载”更精准，边下边播更丝滑

如果你用磁力链接，在获取到元数据之前，客户端根本不知道视频文件的片头在哪里、片尾在哪里。 而用种子文件启动，客户端在一开始就拿到了完美的“地图”。如果你勾选了“按顺序下载”，客户端能极其精准地把前 1% 的带宽全部砸在视频的前 5 分钟数据块上。**缓冲时间从几分钟缩短到几秒钟**。

#### 3. 附赠“加速通讯录”：初始连接速度极快

现在的优质转换站（包括 Magnet2Torrent），在帮你把磁力转成种子时，会自动往 `.torrent` 的文件头里写入它们服务器刚刚帮你探测到的、**当前世界上最快最活的几个 Tracker 服务器地址**。 这就相当于这个种子自带了几个“高效联络员”，你的客户端一打开，不需要盲目去猜谁有资源，顺着文件里自带的 Tracker 拨号，瞬间就能连上几十个活人开始疯狂超车下载。

## Tracker是什么?

- _因为 **BT 下载文件**全都是**其他用户上传给你的**。_
- _所以 **BT 下载速度**就等于**其他用户的上传速度**。_

链接到的做种上传用户越多，你的下载速度就越快，而用户通过 **Peer、DHT、Tracker** 获得。

| 术语          | 说明                                        |
| :---------- | :---------------------------------------- |
| **Peer**    | 在获得一个有效用户后才会起作用，该用户会把它知道的其他用户信息告诉你。       |
| **DHT**     | Peer 加强版，分布式储存用户信息，获得用户的效率更高，但需要下载热门资源来养。 |
| **Tracker** | 记录所有正在上传/下载同一资源的用户信息并提供给你，帮助你与其他用户建立连接。   |
Peer 列表：你的“直接通话通讯录”（活人列表）
Tracker 列表：你的“金牌中介人”（官方花名册）
DHT 列表：你的“江湖朋友圈”（去中心化联络网）

# 在 qBittorrent 里做“流量分流”

机场封 BT，核心抓的不是你的“流量速度（Speed）”**，而是你的**“连接数（Connections）”。

作为开发者，我们很清楚机场（也就是翻墙服务器）的硬件负载逻辑：

- **普通看视频（如 YouTube 4K）：** 你的电脑只会和 YouTube 的几个固定服务器建立 **1 到 5 个 TCP 连接**。虽然下载速度可能高达 10MB/s，但由于连接数极少，服务器处理起来非常轻松（只是纯粹的数据吞吐）。
    
- **BT 下载（即使限速到 100KB/s）：** 按照 P2P 的底层逻辑，你的 qB 会同时向全世界成百上千个 Peer（活人）发起连接请求。这意味着你的电脑会在一瞬间向机场服务器发起 **200 到 500 个并发的 TCP/UDP 连接**。
    

> **致命点就在这里：** 机场服务器（通常是 Linux 系统的 VPS）在同一时间能处理的“总并发连接数”是有系统上限的（受文件描述符 `ulimit` 和内存限制）。 你一个人开了 500 个连接，就会瞬间把这个节点的并发连接坑位占满。导致同节点下的其他几百个用户**连个网页都打不开（提示 DNS 解析失败或连接超时）**。

因此，机场后台的审计脚本（如 Clipy/v2ray 审计规则）通常是这么写的：**“一旦检测到入站请求中包含 BitTorrent 协议特征码，或者单个用户并发 UDP 连接数连续突破 X 个，直接触发断线或封号。”**

**总结：** 哪怕你把下载速度限死在 10KB/s，只要你用它跑传统的 P2P 电影本体下载，它产生的恐怖并发连接数依然会秒触发机场的警报。

qBittorrent（以及比特彗星）专门设计了一个极其聪明的隐藏功能：**“只用代理寻找节点（Tracker/DHT），下载本体走直连”**。

- 打开 qB 的 **设置（齿轮） $\rightarrow$ 连接**。
    
- 找到 **“代理服务器”**（Proxy Server）这一栏。
    
- 类型选择你的本地代理协议（通常是 `SOCKS5` 或 `HTTP`），主机填 `127.0.0.1`，端口填你梯子软件的本地端口（比如 Clash 默认通常是 `7890` 或 `10808`）。
    
- **最核心的两个勾选（关键点）：**
    
    - 🌟 **勾选：仅用于 Peer 连接（仅用代理找人）**
        
    - 🌟 **勾选：将代理用于 Tracker / DHT / 用户交换**
